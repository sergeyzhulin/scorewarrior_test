# Пошаговая симуляция боевых столкновений (ECS + World)

Проект реализует пошаговую симуляцию боёв на клеточной карте для тестирования игровых механик.
Фокус на читаемости, разделении ответственности и расширяемости (добавление новых юнитов/действий/правил без правок ядра).

---

## Ключевые идеи

- **ECS (sparse stores)**: компоненты хранятся в `SparseComponentStore` (dense/sparse), доступ через `EntityManager`. Плотные данные.
- **World**: оркестратор симуляции (фазы тика, порядок спауна, реестры действий/поведений/фильтров).
- **Порядок хода**: игровые объекты обрабатываются в **порядке спауна** (spawn-order), как и требовалось по заданию.
- **Расширяемость через реестры**: действия (actions), поведения (behaviours), фильтры (filters) регистрируются в мире и адресуются через ID.
- **Без внешних зависимостей**: только стандартная библиотека.

---

## Основные сущности системы

### EntityId / EntityManager
- `EntityId` handle (index + generation), позволяет безопасно проверять "жива ли сущность" по средством битовой маски.
- `EntityManager`:
  - `createEntity()` / `destroyEntity()`
  - ленивое создание стора: `store<T>()`
  - делегирующие методы: `addComponent<T>()`, `removeComponent<T>()`, `tryGet<T>()`, `getComponentPtr<T>()`

### SparseComponentStore
- Быстрые операции: `add`, `remove`, `has`, `tryGet`, `getPtr`.
- Уничтожение сущности автоматически удаляет её компоненты во всех поднятых сторах (через `EntityManager`).

### World (симуляция)
- Хранит `GameObject`- список в порядке спауна (`vector<EntityId> + myGameObjectCount`).
- Фазы `tick()`:
  1) Cleanup: удаление терминаторов / чистка "мертвых хэндлов".
  2) Execute: выполнение поведения (один action за тик).
  3) Death evaluation: проверка условий смерти и установка “терминатора”.
  4) Compaction: уплотнение списка объектов при 33% дырок в векторе.

---

## Смерть и условия удаления

- **Отсутствие `DeathCondition`** означает, что объект **бессмертен** (ядро не навязывает HP).
- `DeathCondition.predicateId == 0` — **терминатор**: объект будет уничтожен в фазе cleanup (перед следующими действиями).
- Предикаты смерти регистрируются в `World` и задаются пользовательским кодом (features).

---

## Действия (Actions) и поведения (Behaviours)

### Action
- Интерфейс: `bool tryExecute(World&, EntityId actor) noexcept`.
- Экшен сам решает:
  - может ли выполниться,
  - выбирает цели,
  - выполняет действие и логирует события.

### Behaviour
- Это упорядоченный список `ActionId`.
- В фазе Execute:
  - экшены дергаются по списку,
  - выполнение останавливается после первого `true` (один action за тик).

### Регистрация
- `World::registerAction(Action* action) -> ActionId`
- `World::registerBehaviour(Behaviour* behaviour) -> BehaviourId`
- Lifetime объектов (actions/behaviours) гарантирует вызывающий код (обычно создаются в том же скоупе, где запускается симуляция).

---

## Фильтры взаимодействия (Target-side фильтрация)

Для расширяемых правил "цель модифицирует/запрещает воздействие" реализована схема:

- Если на цели есть `ActionFiltersComponent` (таблица `ActionId -> FilterId`).
- Экшен при выборе таргета:
  1) находит `FilterId` по своему `ActionId`,
  2) вызывает `world.applyFilter(filterId, actor, target, actionId, params)`.

### Type-erasure фильтров
`World` не знает типов параметров (`MeleeParams`, `RangedParams` и т.д.).
Фильтр регистрируется шаблоном `registerFilter<FilterType, ParamsType>(FilterType*)`, а далее применяется через универсальный `applyFilter(...)`.

Это позволяет добавлять новые виды воздействий и новые фильтры в features без правок ядра.

---

## Карта (Map)

- `Map` отдельный класс, не зависит от `World`.
- Хранит **только блокирующую оккупацию**: в клетке может быть максимум **один** blocker.
- Неблокирующие объекты (например, "ворон и/или мина") **не учитываются картой** и могут "сосуществовать" в одной клетке по координатам.

---

## Utilities (features)

В `features` предусмотрены утилиты для простых тестов:
- поиск ближайшей цели по `Coords` компоненте, через проход по dense-стору;
- сбор целей в диапазоне `[minRange..maxRange]`;
- методы для карты (поиск ближайшего блокера, сбор соседей).

---

## Потенциал развития / расширения

1. **Новые юниты и механики**
   Добавляются через новые компоненты + новые actions + новые behaviours, без изменений в Core.

2. **Новые правила взаимодействия**
   Добавляются новыми фильтрами (например, иммунитет к мили, модификация дальности, редирект урона).

3. **Spatial index**
   Линейные сканы по `Coords` легко заменить шардированием на grid-buckets/quadtre без изменения API экшенов.

4. **Более богатая модель карты**
   Можно добавить препятствия, несколько слоёв, стоимость клеток, не меняя основы.

5. **Детерминизм**
   При необходимости можно ввести фиксированный seed RNG и полностью повторяемые прогоны. Это полезно для дебага.

---

## Известные ограничения (кратко)

- Поиск целей/в радиусе сейчас реализован линейным проходом по dense `Coords` (O(N)).
- Lifetime actions/behaviours/filters — ответственность вызывающего кода (в ядре хранится raw pointer).
- Type-erasure фильтров требует корректного соответствия `FilterId` и типа params.
- Карта учитывает только блокирующие сущности; "кто вообще находится в клетке" определяется через координаты.

# **Задача**

Создайте пошаговую симуляцию боевых столкновений юнитов на карте для тестирования игровых механик.

На карте размером W×H клеток можно размещать юнитов.

- Юниты различаются по типам, **действиям** и **характеристикам**.
- Каждому юниту присвоен уникальный идентификатор **Id** (целое число)
- За один ход каждый юнит может выполнить одно действие.
  - Юниты совершают действия в порядке их создания.
- Юнит всегда размещен по каким то координатам на карте.
- Юнит может занимать клетку, не позволяя другим на нее встать.
- Большинство юнитов имеет характеристику **HP** (Health Points).
  - При **HP** ≤ 0 юнит выполняет свои действия в текущем ходу и исчезает перед началом следующего.
- Часть юнитов обладает способностью перемещения:
  - Они могут переместиться в любую из восьми прилегающих клеток за ход.
  - Если цель недостижима за один ход, они продолжают движение к ней в последующих ходах.
  - Статичные юниты должны оставаться неподвижными при любых условиях.

Симуляция заканчивается, когда нет юнитов, способных действовать в следующем ходу, или на карте остается только один юнит.

## Юниты, которых нужно реализовать:

### Мечник
- **Характеристики:** HP, Strength
- Действия
  - **Сокрушающий удар:** Бьет одного случайного юнита в соседней клетке, нанося ему **Strength** единиц урона.
  - Если некого бить, перемещается.

### Охотник
- **Характеристики:** HP, Agility, Strength, Range
- Действия
  - **Стремительный выстрел:** Стреляет в случайного юнита на расстоянии от 2 до **Range** клеток, нанося ему **Agility** единиц урона. Может стрелять только если в соседних клетках нет других юнитов.
  - **Удар из тени:** Если стрелять не может, бьет одного случайного юнита в соседней клетке, нанося ему **Strength** единиц урона.
  - Если некого бить, перемещается.

# Приложение

При запуске приложение получает путь к файлу со сценарием симуляции, где описаны **команды** для создания карты и юнитов, а также перемещения юнитов по карте. 
Приложение должно выводить все происходящие события в **stdout**.

Так как на карте нет препятствий, сложные алгоритмы поиска пути не требуются.

## Команды

- `CREATE_MAP W H` — Создает карту размером `W`×`H`. Эта команда начинает любой сценарий.
- `SPAWN_SWORDSMAN I X Y H S` — Создает мечника с идентификатором `I` в точке `X`,`Y` с характеристиками здоровья `H` и силы `S`.
- `SPAWN_HUNTER I X Y H A S R` — Создает охотника с идентификатором `I` в точке `X`,`Y` с характеристиками здоровья `H`, ловкости `A`, силы `S` и дальности `R`.
- `MARCH I X Y` — Приказывает юниту `I` переместиться в точку `X`,`Y`.

## События

- `MAP_CREATED` — Создание карты.
- `MARCH_STARTED`, `MARCH_ENDED` — Начало и конец перемещения юнита.
- `UNIT_SPAWNED`, `UNIT_DIED` — Создание и смерть юнита.
- `UNIT_MOVED` — Перемещение юнита на клетку.
- `UNIT_ATTACKED` — Атака одного юнита на другого.

Код парсера команд и вывода событий в лог уже содержится в проекте. Если необходимо, его можно менять. Формат ввода и вывода менять нельзя.

Пример сценария находится в файле _commands_example.txt_. Пример лога событий находится в файле _main.cpp_.

При проектировании учитывайте планы по добавлению новых юнитов, действий и характеристик. 

Возможность создания таких Юнитов с таким поведением не трогая код в Core является важным критерием.

## Башня

- **Характеристики:** HP, Power
- Действия
  - **Прицельный выстрел:** Стреляет в случайного юнита на расстоянии от 2 до 5 клеток, нанося ему **Power** единиц урона.
  - Не может перемещаться.

## Лекарь

- **Характеристики:** HP, Spirit
- Действия
  - **Малое исцеление:** Восстанавливает **Spirit** единиц здоровья одному случайному юниту в радиусе 2 клеток.
  - Если некого лечить, перемещается.

## Ворон

- **Характеристики:** HP, Agility
- **Летающий юнит:**
    - Не занимает клетку на карте.
    - Может перемещаться на 2 клетки за ход.
    - Не может быть атакован ближнем боем.
    - Может быть атакован в дальнем бою, минимальная и максимальная дистанция атаки "стрелка" на ворона снижена на 1.
- Действия
  - **Удар когтем:** Бьет одного случайного юнита в соседних клетках, нанося ему **Agility** единиц урона.
  - Если некого бить, перемещается.

## Мина

- **Характеристики:** Power
- Не может быть атакована.
- Не занимает клетку на карте.
- Действия
  - Если в радиусе 2х клеток есть юнит - взрывается в следующем ходу, нанося всем юнитам в радиусе 3х клеток **Power** единиц урона.
  - После взрыва исчезает.
  - Не может перемещаться.
